# Техническое задание на разработку мессенджера Telegram-lite

## 1. Цель документа

Этот документ предназначен для нейросети или команды разработки, которая будет реализовывать проект простого, быстрого и функционального мессенджера по типу Telegram. Документ описывает продуктовые требования, архитектуру, стек технологий, структуру проекта, схему данных, API, realtime-механику, UI/UX, этапы реализации и критерии готовности.

Ключевая цель проекта: создать минималистичный, современный, кроссплатформенный мессенджер с возможностью искать пользователей по нику, обмениваться сообщениями и отправлять изображения. Проект нужен не в виде сырого каркаса, а в виде уже готового к запуску и использованию продукта. Дальнейшие правки, улучшения и расширения функциональности будут вноситься итеративно по мере реального использования приложения.

---

## 2. Краткое описание продукта

Нужно разработать мессенджер с тремя клиентами:

- Web
- Desktop
- Mobile

Приложение должно поддерживать:

- регистрацию и авторизацию пользователей
- профили пользователей
- уникальные username / nickname
- поиск пользователей по username
- личные диалоги один на один
- отправку текстовых сообщений
- отправку фото/изображений
- список чатов
- realtime-обновление сообщений
- статусы сообщений: отправлено, доставлено, прочитано
- отображение онлайн/оффлайн статуса
- минималистичный интерфейс
- светлую и темную тему

Проект должен быть спроектирован так, чтобы его можно было быстро развивать после первой рабочей версии. Первая версия при этом должна быть уже готовым к использованию продуктом, а не недоделанным MVP-шаблоном.

## 2.1 Ключевая установка по формату результата

Нужен не просто прототип, не набор экранов и не частично рабочий MVP-каркас, а уже собранный, связанный и запускаемый продукт, готовый к реальному использованию сразу после сборки и настройки окружения.

Это означает следующее:

- backend должен быть рабочим, а не демонстрационным
- все основные пользовательские сценарии должны быть реализованы end-to-end
- web, mobile и desktop версии должны быть включены в проект
- авторизация, чаты, сообщения, поиск пользователей и отправка изображений должны реально работать
- проект должен запускаться через понятную инструкцию и docker-compose/dev setup
- допускается поэтапная реализация внутри генерации, но итогом должен быть не черновик, а готовая рабочая система
- правки, улучшения и новые функции будут вноситься позже по мере использования продукта

AI-агент не должен ограничиваться созданием архитектурного скелета, моков, заглушек или только части функционала. Нужно собрать полноценную первую рабочую версию продукта.

## 3. Продуктовые требования

### 3.1 Основные сценарии пользователя

Пользователь должен иметь возможность:

1. Зарегистрироваться по email и паролю.
2. Войти в систему.
3. Создать и редактировать профиль.
4. Указать уникальный username.
5. Найти другого пользователя по username.
6. Начать личный чат с найденным пользователем.
7. Отправить текстовое сообщение.
8. Отправить изображение.
9. Просматривать историю переписки.
10. Видеть новые сообщения в реальном времени.
11. Видеть, был ли собеседник онлайн.
12. Видеть статус своих сообщений.
13. Использовать приложение на web, desktop и mobile.

### 3.2 Ограничения MVP

В первой версии не нужно реализовывать:

- групповые чаты
- каналы
- звонки
- видеозвонки
- ботов
- сторис
- сложные реакции
- E2E-шифрование уровня production
- пересылку файлов любых типов, кроме изображений
- голосовые сообщения
- сложную систему прав

Главный акцент первой версии: качественные личные сообщения 1 на 1, быстрый поиск по username, стабильная отправка текста и картинок. Эти функции должны быть доведены до рабочего состояния и реально пригодны для использования, даже если более продвинутые возможности будут отложены на потом.

## 10. Ключевые правила данных

### 10.1 Username

Правила для username:

- обязателен
- уникален
- поиск нечувствителен к регистру
- рекомендуется хранить отдельное поле `username_normalized`
- допустимые символы: латиница, цифры, underscore
- длина: от 4 до 32 символов

Регулярное выражение, рекомендованное для валидации:

```regex
^[a-zA-Z0-9_]{4,32}$
```

### 10.2 Сообщения

- текстовое сообщение может существовать без вложения
- изображение может существовать без текста
- сообщение типа `text_image` содержит и текст, и вложение
- история сообщений должна поддерживать пагинацию

### 10.3 Чаты

Для direct-чата между двумя пользователями должен существовать только один чат.
Нужно гарантировать уникальность direct-чата на уровне бизнес-логики.

## 13. Логика доставки сообщений

### 13.1 Статусы сообщений

Нужно поддержать три состояния:

- `sent` — сервер принял сообщение
- `delivered` — сообщение доставлено на устройство получателя
- `read` — сообщение открыто получателем

### 13.2 Правила

- отправитель видит `sent` после server ack
- когда устройство получателя получает сообщение, сервер фиксирует `delivered`
- когда чат открыт и сообщение просмотрено, фиксируется `read`
- статус не должен откатываться назад

## 14. Логика online / offline

### 14.1 Online status

Пользователь считается online, если:

- у него есть активное websocket-подключение
- соединение валидно и авторизовано

### 14.2 Offline status

При разрыве соединения:

- обновляется `last_seen_at`
- рассылается событие `user:offline` заинтересованным клиентам

### 14.3 Presence storage

Для presence использовать Redis.

Примеры данных:

- active socket ids по user id
- timestamps last activity
- chat typing state

- генерировать собственный storage key
- проверять MIME type на сервере
- предусмотреть антивирусную проверку как future improvement

## 34. Отдельная директива: проект нужен уже готовый

Это критически важное требование.

Нужно не просто начать разработку проекта, а довести его до состояния готовой первой версии, которую можно собрать, запустить и использовать. Подход "сейчас сделаю только основу, а остальное потом" не подходит как итоговый результат.

Допустим поэтапный внутренний процесс разработки, но не допустим незавершенный внешний результат.

Что считается приемлемым итогом:

- проект разворачивается локально по инструкции
- backend работает
- web-клиент работает
- mobile-клиент работает
- desktop-клиент работает
- регистрация и авторизация работают
- поиск пользователей по username работает
- личные чаты работают
- отправка текстовых сообщений работает
- отправка изображений работает

Что не считается приемлемым итогом:

- только архитектурный шаблон
- только backend без клиентов
- только web-версия без остальных платформ
- интерфейс без рабочей серверной логики
- набор моков и заглушек вместо реальных сценариев
- проект, требующий вручную дописывать критические части перед первым запуском

Все следующие изменения, улучшения интерфейса, доработки UX, расширение возможностей и исправления допустимо вносить уже после запуска, по мере реального пользования продуктом.


## 35. Обязательный порядок сдачи результата

AI-агент должен сдавать результат только в следующем порядке:

1. Сначала полностью написать проект
2. Затем выполнить проверку сборки, запуска и основных сценариев
3. Затем найти и исправить все критические и блокирующие баги
4. Затем повторно убедиться, что проект работает после исправлений
5. Только после этого считать первую версию готовой

Недостаточно просто сгенерировать кодовую базу. Нужен именно завершенный цикл: реализация -> проверка -> исправление багов -> повторная проверка.

Правки после этого действительно будут вноситься уже по мере пользования продуктом, но базовая первая версия к моменту сдачи должна быть рабочей, проверенной и исправленной по найденным дефектам.
